About Where's My Bus

A Brisbane transit tracker built out of frustration.

Why I built this

I am sick of buses that are scheduled to arrive on Google Maps just disappearing out of nowhere. You're standing at the stop, the app says two minutes, and then it's gone. No explanation, no cancellation notice, nothing. It just pretends the bus never existed and shows you the next one.

The timetable on Google Maps is very inconsistent, especially when there are a lot of delays. And the new Translink app is rubbish. I just want to know when my bus is actually going to show up.

How it started

This started like many of my projects as a university assessment for CAB432 (Cloud Computing) at QUT. The original version ran on AWS with the whole ecosystem: RDS, ElastiCache, Cognito, S3 etc.

After the unit was completed I decided to migrate it off AWS and self-host it on an old office PC running Proxmox at home. PostgreSQL, Redis, and the Node.js API all run in containers, with the Flask frontend sitting alongside them.

How it works

The backend is a Node.js/Express API pulling from two sources:

  • GTFS static data — Translink's published timetables (stops, routes, trips, shapes) loaded into SQLite.
  • GTFS-RT feeds — Translink's real-time trip updates and vehicle positions, polled every 10 seconds and cached in Redis. Delay data is kept for up to 5 minutes so a brief GPS dropout doesn't throw away a known delay. Vehicle positions expire after 60 seconds.

The frontend is Python/Flask with HTMX for live updates and Leaflet for maps. Timetable and diagram views refresh every few seconds in the background.

I went with Flask and server-side rendering for a couple of reasons. First, I want every stop, route and timetable to have its own URL you can actually navigate to, bookmark, or share. Second, there is a lot of logic involved in processing and displaying the data, and it is just more efficient to do that on the server rather than shipping it all to the browser. HTMX handles the partial page updates without needing a full SPA framework on top of that.

Each row in the timetable shows a colour-coded indicator for how fresh the real-time data is. Green means it just updated, amber means a couple of minutes old, red means it is getting stale. Tap it to see exactly when it was last updated.

Coverage

I originally built this with buses in mind since that is what I use every day, but because it uses the Translink GTFS API it actually works for the whole SEQ network. Trains, ferries, and the G:link light rail on the Gold Coast all show up with real-time data too, which was a nice bonus.
